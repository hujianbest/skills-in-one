# 设计文档模板

若 `AGENTS.md` 为当前项目声明了设计模板，优先使用项目约定。以下为默认结构。

## 默认结构

```markdown
# <主题> 实现设计

- 状态: 草稿
- 主题: <主题>

## 1. 概述
## 2. 设计驱动因素
## 3. 需求覆盖与追溯
## 4. 架构模式选择
## 5. 候选方案总览
## 6. 候选方案对比与 trade-offs
## 7. 选定方案与关键决策
## 8. 架构视图
## 9. 模块职责与边界
## 10. 数据流、控制流与关键交互
## 11. 接口、契约与关键不变量
## 12. 非功能需求与约束落地
## 13. 测试与验证策略
## 14. 失败模式与韧性策略
## 15. 任务规划准备度
## 16. 关键决策记录（ADR 摘要）
## 17. 明确排除与延后项
## 18. 风险与开放问题（区分阻塞 / 非阻塞）
```

## 编写要求

- 区分需求与设计
- 清晰定义边界和职责
- 用追溯视角说明关键需求被哪些模块、流程或接口承接
- 解释为什么所选方案适合该规格
- 至少保留一个紧凑的候选方案对比视图，不让 reviewer 只能从 prose 猜 trade-off
- 测试与验证策略至少说明一条可落到 `hf-test-driven-dev` 的最薄验证路径
- 任务规划准备度要说明哪些边界、接口和风险已经足够支撑 `hf-tasks`
- 说明设计怎样支撑后续任务规划，但不要写成任务计划
- 提供足够支撑任务拆解的细节，但不要写成逐行编码说明

## 保存路径

默认：`docs/designs/YYYY-MM-DD-<topic>-design.md`

若 `AGENTS.md` 声明了设计路径映射，优先使用映射路径。

## 状态同步

设计草稿交评审后，应同步：
- 设计文档状态（`状态: 草稿`）
- `task-progress.md` 中的 `Current Stage: hf-design`
- `task-progress.md` 中的 `Next Action Or Recommended Skill: hf-design-review`
