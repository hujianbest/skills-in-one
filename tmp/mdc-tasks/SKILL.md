# MDC 任务拆解

把需求文档和设计文档转成可执行任务计划。

## 硬性门禁

在任务计划足够清楚之前，不要开始：

- 代码实现
- 无依据地补设计
- 把多个巨大子系统硬塞进一份失真的统一计划

如果缺少需求文档或设计文档，或其中任一文档仍不足以支撑任务拆解，不要假装任务计划已经准备好。

## 目标

最终产出的任务计划至少要回答这些问题：

- 这轮计划到底覆盖哪些范围
- 要先做哪些里程碑
- 每个任务依赖什么
- 每个任务如何验证
- 每个任务完成后算不算真正 done
- 每个任务分别对应哪些需求和设计依据

## 快速工作模式

| 情况 | 当前回合应做什么 | 不要做 |
|---|---|---|
| 需求文档和设计文档都较完整 | 直接整理里程碑、任务、依赖、验证方式和完成定义 | 不要只给一份高层路线图 |
| 只有需求文档，没有完整设计文档 | 明确指出哪些设计缺口会阻塞任务拆解 | 不要臆造模块、接口或数据流后再硬拆任务 |
| 范围过大，包含多个子系统或大平台 | 先给 2-3 种计划切分方式并推荐本轮先写哪一份 | 不要直接写一份覆盖所有子系统的详细总计划 |
| 用户指定了路径、模板或章节骨架 | 严格按用户给定路径和结构起草 | 不要偷偷退回默认路径或默认模板 |
| 用户已有旧任务草稿 | 把它当作草稿输入，只补缺口并保留有效结构、原始标题、状态与已确认里程碑文案 | 不要不分影响面地整份重写 |

优先级规则：

- 只要缺少能支撑任务顺序、依赖或验证方式的关键设计信息，就先停在“还不能可靠拆解”状态
 - 只有当计划边界和关键依赖已经最小稳定时，才进入完整任务拆解

## Workflow

### 1. 先理解最小必要输入

先读完成当前任务拆解所必需的最小上下文：

- 需求文档
- 设计文档
- 用户给定的路径、模板、章节要求
- 旧任务草稿或已有计划（如果用户明确提供）

优先提取：

- 核心用户流程
- 范围内与范围外内容
- 关键模块、接口、数据流与边界
- 约束、外部依赖和风险
- 需求 ID、设计章节号或其他可追溯标识（如果存在）

如果用户提到已有任务草稿，但没有给正文或路径：

- 先索取草稿正文或路径，再进入补稿
- 在未看到草稿前，只能做 intake/triage：说明还缺什么，不能假装已经完成修订

### 2. 先判断是否具备拆解条件

先判断输入是否已经足以支撑“有顺序、有依赖、有验证方式”的任务计划。

以下情况通常说明现在还不能可靠拆解：

- 缺少设计文档
- 关键模块边界仍未定
- 关键接口或数据流未明确
- 需求与设计互相矛盾
- 用户一次要求覆盖多个巨大子系统，但本轮边界还未选定

此时应：

- 回放已确认事实
- 明确列出阻塞项
- 只补充会影响任务拆解的关键问题
- 说明为什么现在不能安全地产出详细计划

不要在这些阻塞项存在时硬写一份看起来完整、但实际上失真的任务计划。

### 3. 必要时先切分计划边界

当范围明显过大时，先给 2-3 种合理切分方式，再推荐一个本轮计划边界。

常见切法包括：

- 按业务子系统拆
- 按主用户流程或 MVP 拆
- 按高风险基础能力和增量业务能力拆

说明每种切法的取舍，然后推荐本轮先落哪一份计划。

如果用户坚持要统一计划，也不要直接给出一份巨细无遗的总表；应先给分阶段的总框架，并明确哪一部分才适合当前回合展开成可执行任务。

### 4. 定义里程碑

把工作整理成能够产生阶段性成果的里程碑。

每个里程碑至少写清：

- 目标
- 包含的任务范围
- 退出标准
- 与前后里程碑的关系

里程碑不是任务列表的替代品。它只负责收口阶段成果，不应取代具体任务。

当输入已经足够完整、且本轮确实进入任务拆解时，优先把里程碑写成可扫描结构，例如表格或字段化块。不要只写“阶段 1 / 阶段 2”这种没有退出标准和前后关系的标题。

### 5. 拆解为可执行任务

任务必须小到可以明确执行和验证。

每个任务至少写清：

- 任务 ID
- 目标
- 对应的需求依据
- 对应的设计依据
- 前置条件或依赖
- 预计会触碰的工件（若当前输入足以判断）
- 验证方式
- 完成定义

若某项内容仍依赖推断，必须显式标成 `[工作假设]` 或 `待确认`，不要伪装成已确认事实。

当输入足够完整时，优先让这些字段在单个任务块中显式可见，而不是零散分布在段落里。优先使用稳定任务 ID，例如 `T1`、`T-APP-1`、`TASK-01`。

优先使用这类任务形态：

- 补齐阻塞实现的缺失信息
- 编写失败测试
- 运行并确认失败
- 实现最小行为
- 运行并确认通过
- 重构并重新验证
- 更新文档、配置、迁移或监控

避免使用这类模糊任务：

- 完成后端
- 完成前端
- 对接系统
- 处理测试
- 后面再补验证

### 5.1 修订旧草稿时的保留规则

如果当前输入包含旧的任务草稿：

- 优先保留原始标题、状态行和已确认里程碑文案，除非它们与当前需求或设计直接冲突
- 若保留原里程碑文案，补稿时仍要把它补齐成可执行计划所需的结构，例如目标、包含任务范围、退出标准和与前后里程碑的关系；若范围说明更适合写在紧邻的“展开说明”里，也要显式写出，不要完全省略
- 若原草稿只包含粗粒度任务，显式说明这些新任务是对哪些原任务的展开，而不是默默换成一份全新计划
- 展开后的每个可执行任务，仍应补齐与完整任务计划同等级的核心字段，至少包括任务 ID、目标、需求依据、设计依据、依赖、预计会触碰的工件（若当前输入足以判断；否则标注为 `待确认` 或 `[工作假设]`）、验证方式和完成定义
- 只补必要缺口与结构，不要顺手把未受影响的章节骨架全部改掉
- 如果必须替换原有内容，要说明替换原因，例如“与当前设计冲突”或“无法支持验证”

### 6. 记录依赖、顺序与风险

任务计划不能只列条目，还要明确它们之间的关系。

至少要说明：

- 哪些任务必须串行
- 哪些任务可以并行
- 哪些外部依赖会阻塞推进
- 哪些风险会影响顺序、范围或验证策略
- 哪些非功能要求需要在任务级落地，而不是最后再统一补

如果需求文档和设计文档中存在冲突，不要静默选择其中一边。应明确指出冲突、给出影响，并把它写成阻塞项或显式工作假设。

### 7. 编写任务计划文档

若用户已经指定文档路径、命名方式或模板，优先遵循。

若没有指定，默认保存到：

- `features/<active>/tasks.md`

若没有现成模板，使用以下最小结构：

```markdown
# <主题> 任务计划

## 背景与输入

## 计划边界

## 里程碑

## 任务拆解

## 依赖关系

## 需求与设计追溯

## 验证策略

## 风险与待确认

## 执行顺序建议
```

写作要求：

- 任务粒度要足够执行和验证
- 任务与需求、设计之间要保持可追溯
- 计划边界要明确，避免把整个大平台一次性写成“看似完整”的总计划
- 验证方式要写在任务级，而不是拖到最后统一补
- 若用户指定了路径和章节骨架，正文必须严格遵循

### 8. 轻量自检

起草完成后，至少检查：

- 每个核心需求是否都有对应任务
- 任务是否真的继承自设计，而不是重新发明设计
- 是否明确写出了依赖和顺序
- 是否存在无法验证的大块任务
- 是否把假设和已确认事实混在了一起
- 范围过大时是否先切分了计划边界
- 是否遵循了用户指定的路径和模板

如果仍有阻塞项，不要假装已经可以直接实现。要明确告诉用户还缺什么。

## 红旗信号

出现以下想法时，先停下。这通常说明你正在为“跳过有效任务拆解”找理由：

| 想法 | 实际要求 |
|---|---|
| “设计还差一点，但先列任务没关系” | 缺设计时不能假装任务顺序和依赖已经清楚。 |
| “先把五个子系统一起排出来，后面再细化” | 范围过大时先切计划边界，而不是硬写一份失真的总计划。 |
| “先写几个大任务，验证和完成定义后面补” | 每个任务都要有验证方式和完成定义。 |
| “需求和设计对应关系实现时自然会知道” | 任务拆解阶段就要写清追溯关系。 |
| “有冲突也先按我猜的继续拆” | 冲突必须显式暴露并处理，不能静默选择。 |
| “旧草稿大概保留意思就行，结构我全部换掉更省事” | 补稿优先保留原始标题、状态、里程碑和未受影响结构，再展开粗任务。 |

## 输出格式

不是每一轮都应该输出“任务计划草稿已完成”。

### A. 当前还不能可靠地完成任务拆解

如果缺少设计文档、关键依赖未定、计划边界未选定，或需求与设计仍存在阻塞性冲突，使用：

```markdown
当前还不能可靠地完成任务拆解。

- 已确认:
  - <已知事实>
- 缺失输入:
  - <缺少的文档、章节或关键信息>
- 仍待确认:
  - <阻塞项>
- 下一步:
  - <只保留一个当前最关键的补齐动作>
```

此模式下：

- 不要假装任务计划草稿已完成
- 不要把大而化之的路线图包装成可执行计划
- 不要输出“可进入实现”

### B. 任务计划草稿已形成

只有在以下条件同时满足时，才使用 completion block：
- 本轮已实际起草、补全或实质更新了一份任务计划草稿
- 文档路径已明确
- 阻塞项、假设和依赖已写明
- 已完成轻量自检

完成时输出应包含：
```markdown
任务计划草稿已完成。

- 文档路径: `<tasks-path>`
- 当前判断: `可进入实现` | `仍需补齐输入`
- 仍待确认:
  - <若无则写"无">

## Ready for tasks-review

任务计划已产出，具备 `mdc-review(review_type=tasks)` 的前置条件：

- [ ] 任务计划已落盘
- [ ] 每个关键任务具备 Acceptance、Files、Verify 字段
- [ ] 依赖关系已明确
- [ ] 追溯覆盖完整
- [ ] 可执行性已确认

推荐下一步：派发 `mdc-review(review_type=tasks)` reviewer subagent。

如需，我可以继续把其中某个里程碑拆得更细，或根据补充输入修订这份任务计划。
```

只有当计划边界稳定、关键依赖明确、任务可验证且不存在阻塞性冲突时，才能判断为 `可进入实现`。

## 反模式

- 把任务计划写成设计文档摘要
- 缺设计时仍伪造详细任务
- 用大到无法验证的任务充数
- 不写依赖关系
- 不写需求与设计追溯
- 把验证留到所有功能都做完以后
- 在大范围场景下直接生成一份失真的总计划

## 完成条件

如果当前回合仍处在输入补齐或范围切分阶段，正确停在“当前还不能可靠地完成任务拆解”并等待更多输入，是允许的中间状态。

只有在以下条件满足时，这个 skill 才算完成：

- 关键输入已足以支撑任务拆解
- 任务计划草稿已落盘
- 已完成一轮轻量自检
- 已明确给出 `可进入实现` 或 `仍需补齐输入` 的判断
