# `mdc-tasks`

## 简介

`mdc-tasks` 用于基于需求文档和设计文档，产出可执行的任务计划。

它关注的不是“再讲一遍设计”，而是把已知需求和设计整理成：

- 里程碑
- 可执行任务
- 依赖关系
- 验证方式
- 完成定义
- 需求与设计追溯

## 目录结构

```text
mdc-tasks/
├── README.md
├── SKILL.md
└── evals/
    ├── README.md
    └── evals.json
```

各文件职责：

- `README.md`：面向使用者的说明文档
- `SKILL.md`：skill 本体
- `evals/README.md`：评测目标说明
- `evals/evals.json`：评测用 prompts

## 适用场景

以下场景适合使用 `mdc-tasks`：

- 已经有需求文档和设计文档，想拆出可执行任务计划
- 当前任务列表过于粗糙，需要补足依赖、验证方式和完成定义
- 想把需求与设计落实到里程碑和任务顺序，而不是直接开始编码
- 范围较大，需要先决定本轮该拆哪一部分任务计划
- 已有一份任务草稿，但还缺依赖关系、追溯信息或验证策略

## 不适用场景

以下情况不建议优先使用这个版本：

- 需求本身还不清楚，当前重点是先澄清 spec
- 设计还没有形成，当前重点是先明确模块、接口和数据流
- 当前目标已经是代码实现、调试或修 bug

如果只有需求、没有完整设计，`mdc-tasks` 不会假装能可靠拆解任务，而会明确指出阻塞项。

## 核心能力

`mdc-tasks` 会做 6 件事：

1. 读取任务拆解所需的最小输入。
2. 先判断当前输入是否真的足以支撑任务拆解。
3. 在范围过大时，先给 2-3 种计划切分方式并推荐本轮边界。
4. 生成里程碑、任务、依赖关系、验证方式和完成定义。
5. 让任务与需求、设计保持可追溯关系。
6. 在结束前做一轮轻量自检，并给出 readiness 判断。

它不会在缺少设计依据时臆造详细任务，也不会把一份高层路线图包装成“已经可执行”的计划。

## 推荐输入

为了让它更高效，建议尽量提供：

- 需求文档或需求摘要
- 设计文档或设计摘要
- 已知的范围边界
- 关键模块、接口、数据流或约束
- 指定的文档路径、模板或章节要求
- 现有任务草稿或已有里程碑清单

如果输入不足，`mdc-tasks` 会先指出阻塞项，而不是强行输出一份详细任务计划。

## 推荐使用方式

最简单的调用方式，是明确告诉它需求和设计都已经具备，并要求拆成任务计划。

```text
我已经有需求文档和设计文档了。请用 mdc-tasks 帮我拆成可执行任务计划，写清里程碑、依赖、验证方式和完成定义。
```

如果你已经知道本轮只想拆某个子系统：

```text
基于现有需求文档和设计文档，只拆审批子系统这一轮的任务计划，不要把公告和消息模块一起展开。
```

如果你希望它按特定路径和模板落盘：

```text
基于现有需求文档和设计文档，任务计划请写到 docs/project-plans/release-approval-plan.md，并使用 概述 / 里程碑 / 任务拆解 / 依赖关系 / 验证策略 这套章节。
```

## 典型工作流

### 1. 初始输入

你先提供需求文档、设计文档，或它们的关键摘要。

### 2. 输入体检

`mdc-tasks` 会先检查：

- 设计是否足够支撑任务拆解
- 需求和设计是否互相冲突
- 范围是否过大
- 是否缺少会影响顺序、依赖或验证方式的关键事实

### 3. 范围切分

如果请求明显过大，`mdc-tasks` 不会直接写一份覆盖所有子系统的详细总计划，而会先给出 2-3 种合理切分方式，例如：

- 按业务子系统拆
- 按主用户流程或 MVP 拆
- 按高风险基础能力与增量业务能力拆

然后推荐本轮先落哪一份计划。

### 4. 任务拆解

在输入足够时，它会输出：

- 里程碑
- 任务清单
- 依赖关系
- 需求与设计追溯
- 验证方式
- 完成定义
- 执行顺序建议

### 5. 轻量自检

文档起草后，它会检查：

- 每个核心需求是否被任务覆盖
- 是否仍然保留了和设计的对应关系
- 是否有大而无法验证的任务
- 是否写清了依赖和顺序
- 是否还有阻塞性待确认项

最后只会给出两种判断之一：

- `可进入实现`
- `仍需补齐输入`

## 输出内容

完成时通常会得到两类结果：

### 1. 任务计划草稿

这是主要产物，会落盘到你指定的路径，或默认路径：

- `docs/tasks/YYYY-MM-DD-<topic>-tasks.md`

### 2. 收尾判断

结尾会包含简短总结，例如：

```markdown
任务计划草稿已完成。

- 文档路径: `docs/tasks/2026-03-31-release-approval-tasks.md`
- 当前判断: `可进入实现`
- 仍待确认:
  - 驳回后是否需要通知抄送人
```

如果输入还不够，它会明确停在“当前还不能可靠地完成任务拆解”，而不是伪装成已经有了可执行计划。

## 常见误区

使用 `mdc-tasks` 时，常见问题包括：

- 只有需求，没有设计，却仍希望直接得到详细任务清单
- 范围明显过大，却仍想一次拆成一份统一总计划
- 任务只写“完成后端”“完成前端”，没有验证方式和完成定义
- 任务与需求、设计之间没有追溯关系
- 有冲突的输入被静默忽略，没有被暴露出来

如果出现这些情况，应该先补齐输入或切小范围，而不是继续硬拆任务。

可以直接复制这些说法来调用：

```text
我已经有需求文档和设计文档了，请先不要写代码，先用 mdc-tasks 帮我产出可执行任务计划。
```

```text
这轮只拆订单审批这一条主流程的任务，不要把整个协作平台都一起展开。
```

```text
如果当前输入还不足以拆任务，请直接告诉我还缺什么，不要硬写计划。
```

```text

```

## 总结

如果你的目标是“基于已有需求和设计，先把执行计划拆清楚，再进入实现”，`mdc-tasks` 就是合适入口。

它最适合任务粒度还不够清晰、依赖和验证方式还没写透、或范围需要先切一刀再落计划的场景。
