# 评审话术示例

这些示例用于规范评审措辞，可根据实际代码调整。

## 示例 1：缺少强断言

场景：

- `RunFoo_BasicCase_Success`

推荐话术：

- 阻塞问题：该测试只检查 API 返回成功，但没有验证可观察输出状态。这意味着即使实现返回 `0` 但结果错误，测试仍可能通过。建议补充对关键输出字段或状态的断言，以真正证明行为正确。

## 示例 2：缺少错误路径覆盖

场景：

- 某个 API 明确会拒绝非法输入，但现有测试只有成功路径

推荐话术：

- 阻塞问题：当前测试集覆盖了正常路径，但没有覆盖非法输入或边界输入。对于存在明确失败行为的 API，这会导致错误处理逻辑未被验证，并提高回归风险。建议根据接口契约补充空值、非法值或边界值场景。

## 示例 3：单个用例过于宽泛

场景：

- 单个测试构造了较大场景，并同时验证多个不相关预期

推荐话术：

- 规范问题：该测试在一个用例体内验证了多个不相关行为，这会增加失败定位成本，也削弱测试意图的清晰度。建议拆分为多个聚焦用例，使每个用例只证明一个行为和一个预期结果。

## 示例 4：高质量 UT 的特征

一个较强的 UT 往往具备：

- 名称清晰表达场景与预期结果
- setup 简短且可读
- 断言与目标行为直接对应
- 在需要时覆盖错误路径或边界场景
- 测试具备确定性且相互独立

推荐话术：

- 未发现阻塞问题。该测试意图清晰，断言与可观察行为直接相关，且用例具备隔离性和可重复性。剩余风险主要在于该测试范围之外的边界场景是否已被其他用例覆盖。

## 强弱对照：同一场景两种写法

### 成功路径只认返回码

弱写法（易制造错误信心）：

- 阻塞问题：`RunFoo_HappyPath` 仅断言 `EXPECT_EQ(ret, 0)`，未检查任何输出或状态字段；实现若返回成功但写错内部状态，测试仍会通过。

强写法（话术方向）：

- 建议在通过返回码的同时，至少断言契约规定的关键输出字段或不变量；若契约未文档化，先写明假设再建议与接口方对齐可观察行为。

### 失败路径未约束状态

弱写法：

- 阻塞问题：`RunFoo_NullPtr_Fails` 只检查 `ret != 0`（或固定错误码），未说明输出对象是否应保持不变。

强写法：

- 若假设失败不应修改用户可见状态，应增加“调用前后关键字段一致”或“明确错误码 + 无副作用”的断言；若契约未定义，应标出“契约缺口”而非武断要求。

### 多字段用一条弱不等式带过

弱写法：

- 规范或阻塞（视业务而定）：用 `EXPECT_GT(a+b+c, 0)` 代替对 `a`、`b`、`c` 各自应有形态的断言，失败时难以定位且易漏检单字段错误。

强写法：

- 分字段断言或对照契约给出期望值；与算法版本绑定的 golden 值需在评审中说明可维护性权衡。
