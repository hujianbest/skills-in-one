# UT 审查准则

用于判断单元测试是否合理、是否符合规范。与具体测试框架无关；C++ GoogleTest 的宏、fixture、参数化等细节见 `gtest-cpp.md`。

## 1. 测试意图

检查：

- 用例名是否表达了场景和预期结果。
- 一个用例是否只验证一个连贯行为。
- 不需要反向推导复杂 setup，也能看出测试到底在验证什么。

常见问题：

- 名称过于模糊，例如 `BasicTest`、`SuccessCase`。
- 一个用例混合了多个不相关行为。
- setup 过重，掩盖了真正的测试目标。

## 2. 断言质量

检查：

- 断言是否验证了对外有意义的行为。
- 一旦行为回归，测试是否真的会失败。
- 断言是否足够具体，能抓住预期问题。

常见问题：

- 完全没有断言。
- 只检查返回码，没有检查关键状态变化。
- 只断言 mock 被调用，但没有验证最终行为。
- 在可以做更强断言时，仍只写了 `非 0`、`非空`、`未抛异常` 一类弱断言。

## 3. 覆盖完整性

检查：

- 是否覆盖正常路径。
- 是否覆盖关键边界条件。
- 在有明确失败行为时，是否覆盖非法输入或错误路径。
- 是否覆盖高风险分支。

常见问题：

- 只测试成功路径。
- 未覆盖空值、非法值、越界值或边界输入。
- 容易回归的分支没有被触达。
- 对参数敏感的逻辑，测试矩阵过窄。

## 4. 隔离性与确定性

检查：

- 测试结果是否不依赖执行顺序。
- 是否避免依赖真实时间、随机数、全局状态、环境状态或共享可变 fixture，除非这些因素已被控制。
- 外部依赖是否在合适边界被隔离或替身化。

常见问题：

- 隐式依赖前置测试或共享状态。
- 时间或随机行为未受控。
- 在单元测试中直接依赖真实文件系统、网络、设备或进程状态。
- mock 不完整，仍残留不确定行为。

## 5. 可维护性

检查：

- Arrange、Act、Assert 是否清晰。
- 测试数据是否尽量精简且足够表达场景。
- 辅助函数是否减少重复，同时没有隐藏意图。
- 常量和 fixture 是否让场景更易读。

常见问题：

- 大段 setup 让被测行为不清晰。
- 大量复制粘贴，只改了少量但不明显的参数。
- 出现没有解释的魔法值。
- 辅助函数隐藏了关键 setup 细节。

## 6. 规范符合性

检查：

- 命名是否符合项目约定。
- 测试文件位置与分组是否符合项目结构。
- fixture、参数化测试、mock 的使用方式是否一致。
- 测试范围是否仍属于"单元测试"，而不是混入集成测试行为。

常见问题：

- 测试放在错误的 suite 或错误的层级。
- 命名没有体现输入场景和预期结果。
- 为很窄的单元行为引入过重 setup。
- 把集成式验证放进 UT，但没有合理说明。

## 7. 用例编写规范

### 7.1 用例的通用要求

#### 7.1.1 【规则】禁止写出永远成功的测试用例

这种测试用例没有assert，一般是用来完成代码覆盖率指标的的，用例没有检查业务代码是否真的保证成功，但是测试报告却给出了虚假的信息。

#### 7.1.2 【建议】每个用例应该是单一职责的，只测试一个场景，用例里面禁止使用switch-case、if/else等。

#### 7.1.3 【规则】用例之间应该是独立的，用例之间不能相互调用，用例之间调用不能有调用顺序。

#### 7.1.4 【规则】用例应该是可以重复执行的，不受外界环境的影响

#### 7.1.5 【规则】不能为了测试添加相关的业务代码，如果有，也是和业务代码物理隔离的。

#### 7.1.6 【规则】测试中禁止使用printf/println/std::cout/System.out等方式输出后进行人工判断，必须使用assert来进行自动验证。

#### 7.1.7 【规则】IT测试组件/模块的接口，UT测试类的接口

单元测试应该对公有API进行测试，私有方法的测试应该由公有方法来覆盖。原因如下：

- 私有方法是不对外暴露，在用例中通过高级手法调用私有方法，打破类的封装。
- 私有方法在重构时是不稳定的，入参、返回值及方法名都是不断变化的。

#### 7.1.8 【规则】合理使用Test Double（测试替身）

Test Double包括Dummy, Spy, Stub, Mock, Fake，主要使用Stub和Mock，Mock无法复用但使用简单，Stub稍复杂但考虑尽量复用。

什么情况下才能使用测试替身：

- 硬件依赖
- 现实环境中难以产生的输入
- 慢速对象 —— 网络，数据库等等
- 对于易变事物的依赖 —— 时钟
- 对于尚未开发完成事物的依赖
- 测试对象外的接口，如对其它组件/模块的调用。

过量使用Mock会带来：

- 重构时对于Mock的大量修改带来的成本
- Mock对象的非真实性带来测试有效性的降低

#### 7.1.9 【建议】编写assert时建议包括错误信息

```C++
public void should_not_equals_when_compare_tow_strings() {
    assertEquals("foo", "bar");    //不推荐
    // 推荐使用下面的形式:
    assertEquals("Foo does not equals bar", "foo", "bar"); //推荐
}
```

#### 7.1.10 【规则】所有以类为测试对象的测试用例类命名为："<被测试类的类名>+Test"

## 8. 结论判定

结论可按以下逻辑给出：

- `通过`：没有阻塞问题，剩余问题较轻。
- `需改进`：核心行为已覆盖，但在覆盖、断言或规范方面存在明显不足。
- `不通过`：测试会制造错误信心、漏掉关键行为，或本身不是可靠的单元测试。

## 问题撰写方式

每条问题尽量包含：

1. 问题是什么。
2. 为什么重要。
3. 更强的测试应如何体现。

示例：

- `RunFoo_InvalidInput_Fails` 只检查了返回码，没有断言输出状态保持不变。这会漏掉"失败时发生部分写入"一类问题，并对错误处理形成错误信心。应补充对失败后可观察状态的断言。
