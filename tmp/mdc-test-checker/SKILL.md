# MDC Test Checker

`mdc-test-checker` 是一个面向测试质量审查的通用 Skill。使用它时，应把任务理解为“风险导向的测试评审”，而不是样式检查。

## Skill 名称

`mdc-test-checker`

## Skill 描述

`mdc-test-checker` 用于审查单元测试用例是否合理、覆盖是否充分、断言是否有效、是否稳定可维护以及是否符合团队规范。

## 何时触发

当用户在做以下事情时，直接使用 `mdc-test-checker`：

- 审查 UT / 单测 / unit test
- 判断测试是否合理、是否符合规范
- 识别弱断言、边界缺失、错误路径缺失
- 点评 gtest / GoogleTest / pytest / JUnit 等测试代码质量
- 评估某批测试是否会制造“错误信心”

如果请求明显是在“看测试是否写得好”，即使用户没有明确说“请用 Skill”，也应触发。

## 不适用本 Skill 的场景

以下情况不要强行套用 `mdc-test-checker` 的全套评审输出，可简短说明后按用户真实任务处理：

- 只做性能压测、容量规划或 profiling，不涉及单测设计评审
- 集成测试、端到端测试的环境与数据问题排障，而非 UT 代码评审
- 用户只要求“实现/补全测试代码”，且未要求评审质量
 - 纯格式化、命名风格或 linter 规则讨论，且未涉及断言与覆盖

## Workflow

### 1. 收集测试上下文

准备评审所需的基本信息：

- 确认被测行为和目标测试文件路径
- 识别测试用例对应的源文件
- 确定测试框架（gtest、pytest、JUnit 等）
- 若有公开 API 或接口声明，优先阅读
- 读取必要的 reference 文档：
  - `references/review-rubric.md`（通用评审准则）
  - `references/gtest-cpp.md`（C++ GoogleTest 专项，如适用）
  - `references/review-examples.md`（评审措辞参考，可选）

### 2. 识别测试用例

对每个测试用例进行分析：

- 识别用例名字（描述测试的行为）
- 识别被测的函数/模块
- 识别测试数据（输入、预期输出）
- 识别断言类型和强度
- 识别测试范围（正常路径、边界、异常）

### 3. 评审测试保证性

判断测试是否"真的在测试行为"而不仅是"跑通代码"：

**关键问题：**
- 这个测试在验证什么可观察行为？
- 如果实现错了，这个测试能抓住吗？
- 测试依赖实现细节还是行为契约？
- 测试是否只验证了"能跑过"？

### 4. 检查断言质量

评估断言的有效性和强度：

**常见弱断言模式：**
- 只有返回码检查（如 `EXPECT_EQ(ret, 0)`）
- 没有验证输出或状态变化
- 注释说明应该是什么，但断言未验证
- 断言过于宽泛（如 `EXPECT_TRUE(value > 0)`）

**建议：**
- 断言应直接表达期望结果
- 复杂对象应验证关键字段
- 失败应产生清晰错误信息

### 5. 评估测试覆盖

检查测试覆盖的场景：

**必需覆盖：**
- 正常路径（happy path）
- 边界条件（空值、最小值、最大值）
- 错误路径（异常、失败输入）
- 典型错误案例

**常见缺失：**
- 只测成功路径
- 缺少边界测试
- 异常路径未验证
- 资源泄漏场景未测试

### 6. 检查测试隔离性和稳定性

检查测试是否可独立、稳定运行：

**常见问题：**
- 依赖共享全局状态
- 依赖执行顺序
- 未清理资源
- 硬编码时间戳或随机性
- 依赖外部系统

**要求：**
- 每个测试独立可运行
- 任意顺序运行结果一致
- 有适当的 setUp/tearDown
- 测试数据自包含

### 7. 评审可维护性

评估测试的长期可维护性：

**可维护性标志：**
- 测试名称清晰描述行为
- 避免重复的测试设置代码
- 使用参数化测试避免复制粘贴
- 有意义的测试数据命名
- 适当的注释说明复杂场景

### 8. 生成评审报告

使用标准模板输出评审结果：

```markdown
## 审查结果

### 总体结论
- 通过 / 需改进 / 不通过

### 阻塞问题
- [文件路径:用例名] 问题描述及建议

### 规范问题
- [文件路径:用例名] 命名、结构问题

### 覆盖评估
- 已覆盖：[场景列表]
- 缺失：[场景列表]

### 改进建议
- [具体改进建议]

### 结论
- [1-2 句总结]
```

**输出要求：**
- 每条问题追溯到具体用例（文件路径+用例名）
- 优先链阻塞问题（不是格式问题）
- 问题说明应包含：是什么、为什么重要、如何改进
- 评审可复核、可追踪

## 如何审查

`mdc-test-checker` 按下面顺序做判断：

1. 先识别每个用例到底在验证什么行为。
2. 判断该用例是否真的证明了行为，而不只是跑通代码。
3. 检查断言强度、边界覆盖、错误路径、隔离性和可维护性。
4. 优先指出会制造错误信心的问题，例如：
   - 只有返回码断言
   - 只测成功路径
   - 依赖共享状态或执行顺序
   - 只验证实现细节，不验证可观察结果

## 信息不足时的处理

在缺少接口契约、错误码约定或“失败时是否应保持状态不变”等上下文时：

1. 先用一两句话写清**你采用的假设**（例如“假设失败路径不应修改输出对象状态”）。
2. 再在该假设下给出评审结论与建议。
3. 不要无假设地把猜测写成确定事实。

## 证据要求

- 每条**阻塞问题**必须能追溯到具体对象：至少写出**测试文件路径**和**用例名**（或等价可定位描述）；必要时用一句话概括相关断言在测什么。
- **规范问题**同样尽量带来源，便于作者修改。
- 避免只有泛泛结论而没有可定位条目。

## 输出要求

使用直接、可执行、偏 reviewer 风格的表达，让 `mdc-test-checker` 的输出稳定、可复核、可追踪。

- 先给总体结论，再列问题。
- 优先写行为风险，不要先写格式问题。
- 每条问题尽量说明：问题是什么、为什么重要、建议如何补强。
- 如果没有发现问题，明确写“未发现阻塞问题”，并补充剩余风险。
- 除非用户要求，不要长篇讲测试理论。

## 默认输出模板

除非用户指定其他格式，否则使用下面结构（须保留各级标题以便抽检）：

```markdown
## 审查结果

### 总体结论
- 通过 / 需改进 / 不通过

### 阻塞问题
- [文件路径 + 用例名] 问题是什么，为什么重要，建议如何补强

### 规范问题
- [文件路径 + 用例名] 命名、结构或约定问题

### 覆盖评估
- 已覆盖：...
- 缺失：...

### 改进建议
- ...

### 结论
- 用 1 到 2 句话总结该批 UT 是否合理、是否符合规范。
```
