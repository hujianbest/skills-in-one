# mdc-test-checker 最小验收清单

用于人工或半自动抽检本 Skill 是否达到计划中的验收标准。可与 `evals/evals.json` 的输出、`evals/trigger-eval.json` 的触发预期对照使用。

## 1. 触发

- 对 `trigger-eval.json` 中 `should_trigger: true` 的 query，助手应进入 UT 评审语境（结构化结论或明确按审查流程执行）。
- 对 `should_trigger: false` 的 query，助手不应默认套完整「审查结果」八股模板；除非用户同时明确要求评审测试质量。

## 2. 输出结构

- 对 `evals/evals.json` 中的正向评测 prompt，回复中须出现以下**标题或等价结构**：`总体结论`、`阻塞问题`、`规范问题`、`覆盖评估`、`结论`（或用户约定的等价分段）。
- `总体结论` 中须出现「通过」「需改进」「不通过」之一或清晰同义表述。

## 3. 质量与可追溯性

- 至少一条「阻塞问题」或「规范问题」须包含**可定位信息**：文件路径、用例名（或 `TEST_F` 名），或等价描述。
- 「阻塞问题」应优先于纯命名、缩进等样式建议出现；若仅有样式问题，总体结论不应写成「通过」除非明确声明无行为风险。

## 4. 信息不足

- 若缺少 API 契约，回复中须出现**假设**表述，再给出建议，而非把猜测写成确定事实。

## 5. 抽检频率建议

- Skill 文案或 `references/` 有重大变更后：跑一遍 `evals.json` 全部 prompt + 抽查 `trigger-eval.json` 中 false 各一条。
