# 架构设计评审检查清单

> **说明**：`mdc-arch-design` 产出两份文档——过程交付件 `arch-design-record.md` 和项目长期资产 `component-design-doc`（`docs/<component-name>-design.md`）。本检查清单覆盖两份文档的全部评审维度。
>
> **评审性质**：本评审是**组件级一次性门禁**——仅在新组件首次设计时执行。通过后，`component-design-doc` 成为该组件的**设计基线**，下游所有 AR 实现设计（`mdc-ar-design`）须以此为基线描述增量变更；已有基线的组件不再重复此评审。

## 评审维度

| ID | 维度 | 对应产出 | Pass Condition |
|---|---|---|---|
| `AD1` | 需求覆盖与追溯 | arch-design-record + component-design-doc | 设计覆盖关键需求，新增组件可回指到已批准规格，功能列表覆盖规格需求 |
| `AD2` | 架构一致性 | arch-design-record | 边界、职责、视图清楚 |
| `AD3` | 决策质量 | arch-design-record | 至少比较两个方案，trade-offs 显式 |
| `AD4` | 约束与 NFR 适配 | arch-design-record | NFR 落到具体模块 |
| `AD5` | 接口与组件定义 | arch-design-record + component-design-doc | 新增模块职责、接口定义清晰，接口声明了并发约束 |
| `AD6` | 测试准备度 | component-design-doc | 测试设计覆盖功能点，观测点明确，成本评估已填写 |
| `AD7` | 实现设计完整性 | component-design-doc | 代码结构、实现模型、数据设计、运行视图、软件单元设计完整 |
| `AD8` | 文档合规性 | component-design-doc | 覆盖模板所有必需章节，无模板占位符残留 |

### `AD5` 接口与组件定义（arch 级特化）

- 新增模块是否明确定义职责和边界？
- 关键接口契约是否足够清晰？
- 接口是否声明了并发约束（支持/不支持 + 保护机制）？
- 模块间依赖关系是否合理？
- 是否足以支撑下游 `mdc-ar-design`？

### `AD7` 实现设计完整性

- 代码结构模型是否与实际 CMake/目录一致？
- 实现模型类图是否与接口定义一致？
- 数据设计是否覆盖组件内部数据结构？
- 运行视图是否覆盖交互/并发场景？
- 软件单元设计是否细化到函数级？
- **每个功能点的每个场景是否都有 PlantUML 时序图？** 时序图必须细化到软件单元级别的交互（参与者包括具体类/软件单元，消息包含方法调用），而非仅组件级粗粒度交互
- **各章节内容是否足以支撑后续各 AR 的实现设计？** 逐章检查：
  - 功能列表 → 各 AR 可按功能编号追溯并圈定范围
  - 接口定义 → AR 可直接引用接口契约进行实现
  - 实现模型（类图）→ AR 可按类归属拆分实现任务
  - 功能详设时序图 → AR 可按场景流程直接编写实现代码
  - 软件单元设计 → AR 可按函数级设计编写代码

## Anti-Pattern 检测

| ID | Anti-Pattern | 检测信号 | 正确做法 |
|---|---|---|---|
| `AA1` | 无 NFR 评估 | 只看功能，忽略性能/安全/可扩展 | 必须评估 NFR |
| `AA2` | 只审 happy path | 不审查失败路径 | 审查失败模式、重试、熔断 |
| `AA3` | 无权衡文档 | 隐藏 trade-off | 显式记录权衡及理由 |
| `AA4` | 单点故障未记录 | 系统有隐性脆弱 | 映射所有 SPOF，要求缓解 |
| `AA5` | 组件定义空洞 | 架构设计没定义新增组件职责/接口 | 本 skill 是阻塞门，明确定义 |
| `AA6` | 实现设计缺失 | 只产出 arch-design-record，未产出 component-design-doc | 两份文档都必须产出 |
| `AA7` | 接口缺并发约束 | 接口定义未声明是否支持并发调用 | 每个接口必须声明并发约束 |
| `AA8` | 测试设计空洞 | 测试项无观测点或仅写"功能可用" | 每个测试项必须有明确观测点 |
| `AA9` | 时序图缺失或粗粒度 | 功能详设无时序图，或时序图参与者仅为组件级（如"A→B: 处理"），未细化到软件单元/类的方法调用 | 每个功能点的每个场景必须有 PlantUML 时序图，参与者细化到类/软件单元，消息体现方法调用和参数 |
