# AR 设计评审检查清单

## 评审维度

| ID | 维度 | Pass Condition |
|---|---|---|
| `AR1` | 需求覆盖与追溯 | 每个功能点可回指到规格/架构设计的具体条目，无孤儿功能点 |
| `AR2` | 功能点完备性 | 功能点分解覆盖所有需求，无遗漏，优先级明确，粒度适中（单个功能点可独立测试） |
| `AR3` | 实现设计充分性 | 开发者拿到文档可直接编码，不需二次猜测；实现思路包含关键设计要点 |
| `AR4` | 接口契约清晰度 | 每个接口有完整的参数名/类型/描述/返回值，错误码/异常返回有定义 |
| `AR5` | MDC 场景覆盖 | 并发、启动退出、休眠唤醒、可靠性、SELinux 五大场景均有分析，"否"需给出理由 |
| `AR6` | 异常流程完备性 | 正常流程 + 异常流程均有设计，不是只有 happy path；关键异常有处理策略 |
| `AR7` | 测试设计可执行性 | UT/接口测试/异常场景测试覆盖功能点，逻辑覆盖度明确，测试点可追溯到功能点 |
| `AR8` | 代码设计可落地性 | 包结构、类设计、类图与实现对应，命名符合规范，修改范围明确 |
| `AR9` | 基线一致性 | AR 设计与组件基线保持一致：引用基线正确、功能 ID 对齐、接口变更为增量式 |

## `AR1` 需求覆盖与追溯（详细检查项）

- 每个功能点是否可回指到规格中的 FR/NFR 条目或架构设计中的组件/接口？
- 是否存在无来源的"凭空"功能点？
- 是否存在规格中有但功能点分解中遗漏的需求？
- 若涉及架构设计，AR 设计是否在架构边界内实现，未越界？

## `AR2` 功能点完备性（详细检查项）

- 功能点分解是否覆盖了需求描述中的所有能力？
- 每个功能点是否有清晰的名称、描述和优先级？
- 单个功能点是否可独立测试？若多个功能点耦合在一起无法独立验证，说明粒度过粗
- 优先级标注是否合理？不是所有功能点都是 high

## `AR3` 实现设计充分性（详细检查项）

- 实现思路是否回答了"从规格到代码"的关键路径？
- 是否明确标注了代码重构需求、新增/修改类清单、业务流程、现有功能影响、代码移植？
- 流程图是否覆盖了核心业务场景的完整路径？
- 时序图是否准确反映了模块间交互，参与者命名是否与代码设计一致？
- 用例图（如有）是否覆盖了 AR 涉及的操作场景？

## `AR4` 接口契约清晰度（详细检查项）

- 内部接口：参数名/类型/描述/返回值是否完整？
- 外部接口（服务器/SDK）：是否引用了接口定义文档地址？
- 接口变更：是否说明了与现有接口的关键变更点？
- 缺失接口定义或参数描述含糊（如"相关信息"、"配置数据"）→ 不通过

## `AR5` MDC 场景覆盖（详细检查项）

- **并发场景**：是否分析了线程模型和数据保护？如果回答"否"，是否给出了不涉及并发的理由？
- **启动退出**：是否分析了进程启动/退出对本 AR 的影响？
- **休眠唤醒**：是否分析了休眠唤醒场景下的影响和处理策略？
- **可靠性**：是否分析了与周边模块/硬件交互的可靠性设计（启动时序、依赖关系）？
- **SELinux**：是否明确了进程需要触碰的系统部分（可执行文件路径、配置文件路径、IPC路径）？
- MDC 场景中"否"的回答缺少理由说明 → 需修改

## `AR6` 异常流程完备性（详细检查项）

- 正常流程描述是否完整、步骤清晰？
- 异常流程是否至少覆盖：外部依赖断链/超时、参数非法、资源不可用、状态不一致？
- 关键异常场景是否有明确的处理策略（重试/降级/告警/默认值）？
- 只有 happy path 没有 exception path → 不通过

## `AR7` 测试设计可执行性（详细检查项）

- 测试点汇总是否覆盖了所有功能点？
- 逻辑覆盖程度是否明确（语句覆盖/判定覆盖/路径覆盖）？
- 单元测试：是否列出了关键功能的测试场景和预期结果？
- 接口测试：是否按接口定义覆盖了必选字段、可选字段、字段长度校验？
- 异常场景测试：是否覆盖了与外部交互的断链/超时/部分失败？
- 测试脑图（如有）是否与测试设计内容一致？
- 测试用例可以追溯到功能点，无悬空测试

## `AR8` 代码设计可落地性（详细检查项）

- 包结构设计：包路径、描述、包含类是否完整？
- 类设计：类名、职责、公开 API 是否清晰？
- 类图：是否与类设计描述一致？关系（继承/组合/依赖）是否正确？
- 命名规范：变量/函数小驼峰，类名大驼峰，常量小驼峰——是否有违反？
- 修改范围：是否明确标注了新增/修改哪些文件？
- 是否存在设计泄漏到规格层的内容（不应在 AR 中重新定义业务规则）？

## `AR9` 基线一致性（详细检查项）

**基线来源判断**：根据 `mdc-ar-design` 步骤 1.1 的三种基线来源，检查项不同：

**场景 A：component-design-doc 存在（`docs/<component-name>-design.md`）**
- AR 文档是否明确引用了基线文档？引用路径是否正确？
- AR 功能点 ID 是否与基线中功能列表 ID 对齐？是否存在基线中无对应的功能点？
- AR 中的接口变更是否为增量式（基于基线接口的扩展/修改），而非重新定义？
- AR 实现设计是否在基线定义的架构边界内，未越界定义新组件或新系统接口？

**场景 B：组件代码存在但无设计文档**
- AR 文档是否基于逆向分析描述了现有代码结构？
- AR 设计是否与现有代码的类结构、包结构、接口定义一致？
- 新增/修改内容是否在现有代码架构内，未引入与现有结构矛盾的变更？

**场景 C：无基线且无代码 → 红旗**（应回退到 mdc-arch-design，不应直接进入 AR 设计）

## Anti-Pattern 检测

| ID | Anti-Pattern | 检测信号 | 正确做法 |
|---|---|---|---|
| `RA1` | 功能点空洞 | 功能点描述为空或仅复述需求标题 | 功能点必须有具体的行为描述 |
| `RA2` | 实现思路缺失 | 4.1 章节只写"按需求实现"无关键设计要点 | 必须包含实现路径、影响评估、代码变更范围 |
| `RA3` | 接口参数含糊 | 接口参数类型为"Object"或描述为"相关数据" | 参数必须有具体类型和语义描述 |
| `RA4` | MDC 场景全"否" | 五大 MDC 场景全部标记为不涉及且无理由 | 至少对"否"给出判定依据 |
| `RA5` | 只有 happy path | 流程说明只有正常流程无异常流程 | 必须覆盖异常场景 |
| `RA6` | 测试与功能点脱节 | 测试设计无法追溯到功能点 | 测试点必须可映射到功能点 |
| `RA7` | 架构级决策在 AR 中 | AR 中出现新增组件/模块/系统接口定义 | 架构级决策应回退到 mdc-arch-design |
| `RA8` | 设计与代码脱节 | 类图/类设计与后续代码变更描述不一致 | 设计和代码变更必须一致 |
| `RA9` | 缺少修订记录 | 文档无修订记录或修订记录不完整 | 必须有完整的修订记录 |
| `RA10` | 模板占位符残留 | 文档中残留 {DATE}、{AUTHOR}、TBD 等占位符 | 所有占位符必须替换为实际内容 |
| `RA11` | 基线存在但未引用 | component-design-doc 存在但 AR 未引用或引用路径错误 | AR 必须显式引用基线文档并按基线 ID 追溯 |
| `RA12` | 无基线凭空创造 | 无 component-design-doc 且无组件代码，AR 却直接设计实现 | 应回退到 mdc-arch-design 先建立组件基线 |
| `RA13` | AR 越界重定义组件架构 | AR 中出现与基线矛盾的组件划分/接口重定义/模块职责变更 | 架构级变更须回退到 mdc-arch-design |

## 评分规则

9 维度评分（0-10）：

| 维度 | 对应 ID | 权重 | <6 后果 |
|---|---|---|---|
| 需求覆盖与追溯 | AR1 | 关键 | 阻塞 |
| 功能点完备性 | AR2 | 关键 | 阻塞 |
| 实现设计充分性 | AR3 | 关键 | 阻塞 |
| 接口契约清晰度 | AR4 | 关键 | 阻塞 |
| MDC 场景覆盖 | AR5 | 重要 | 需修改 |
| 异常流程完备性 | AR6 | 重要 | 需修改 |
| 测试设计可执行性 | AR7 | 重要 | 需修改 |
| 代码设计可落地性 | AR8 | 重要 | 需修改 |
| 基线一致性 | AR9 | 关键 | 阻塞 |

**关键维度 < 6 不得通过**，重要维度 < 6 需修改。

## Finding Classification

沿用 mdc-review 统一规则：

- `USER-INPUT`：需要用户/业务方确认的业务规则、优先级、接口契约细节
- `LLM-FIXABLE`：可在不新增业务事实的前提下由 mdc-ar-design 直接修的问题（补充描述、调整命名、补充异常流程、对齐测试与功能点等）

## Verdict Guidance

- `通过`：所有关键维度 >= 6，重要维度 >= 6，无阻塞 USER-INPUT finding
- `需修改`：AR 有用且方向正确，1-2 轮定向回修可达标
- `阻塞`：核心功能点遗漏、实现思路完全缺失、存在架构级决策需回退到 mdc-arch-design
